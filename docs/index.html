✦ Self-hosted SSH Manager

Reconnect

A browser-based SSH manager that lets you connect to remote servers, run commands, manage files, and monitor system stats — all without installing a native client. Open a tab, log in, and you're in.

AES-256-GCM Encryption WebSocket PTY Monaco Editor Self-hosted Responsive

🖥️ Remote Server
Your SSH target
  • → Browser → Express: REST API for CRUD operations; Server-Sent Events for streaming command output; WebSocket for interactive PTY.
  • → Express → SSH: The ssh2 library manages connections, SFTP channels, and PTY allocation.
  • → SQLite: Stores server configurations (encrypted credentials), saved commands, and bookmarked file paths via better-sqlite3.

Getting Started

Prerequisites

  • ✓ Node.js 18 or later
  • ✓ An SSH server to connect to

Installation

bash# Clone the repository
git clone <repo-url>
cd reconnect

# Install dependencies
npm install

# (Optional) Set an application password
echo "APP_PASSWORD=your-secret" > .env

# Start the server
npm start
# → Listening on http://127.0.0.1:9898
💡 Development tip: If APP_PASSWORD is not set in your .env, the application skips the login screen entirely and loads the main UI directly — just npm start and go. Set APP_PASSWORD before exposing the app to any network.

⚡ Auto-Start with PM2

PM2 keeps Reconnect running in the background and restarts it automatically after crashes or reboots.

CommandWhat it does
pm2 statusSee if Reconnect is running
pm2 logs reconnectTail live logs
pm2 restart reconnectRestart the server
pm2 stop reconnectStop the server
pm2 start npm --name reconnect -- startStart it again if stopped
🚀 First-time setup — run once to configure PM2:
npm install -g pm2
pm2 start npm --name reconnect -- start
pm2 save
pm2 startup  ← run the printed sudo command to enable auto-start on login

🌐 Optional: Custom Local Domain

Access Reconnect at http://reconnect.zoho.tool:9898 instead of the default http://127.0.0.1:9898. This is a one-time setup — no code changes required.

Step 1: Add the hosts entry

macOS — Terminalecho "127.0.0.1 reconnect.zoho.tool" | sudo tee -a /etc/hosts

Step 2: Verify it works

bashping -c 1 reconnect.zoho.tool
# Should show: 127.0.0.1

Step 3: Open in browser

URLhttp://reconnect.zoho.tool:9898
🔖 Bookmark this URL. Because .tool is not a standard public TLD, some browsers may treat a bare reconnect.zoho.tool as a search query — always use the full http:// prefix.

To undo (remove the custom domain)

bashsudo nano /etc/hosts
# Find and delete the line: 127.0.0.1 reconnect.zoho.tool
# Save with Ctrl+O, exit with Ctrl+X

🔌 Step 4 (advanced): Access without a port number

Want http://reconnect.zoho.tool with no :9898? Three approaches — pick whichever suits you.

Option A — macOS pfctl port forwarding (recommended)

Keeps the app on 9898. The OS silently forwards port 80 → 9898. No PM2 changes needed.

macOS — Terminalsudo pfctl -e 2>/dev/null; echo "rdr pass on lo0 inet proto tcp from any to 127.0.0.1 port 80 -> 127.0.0.1 port 9898" | sudo pfctl -ef -

What the output means

You may see messages like these — all of them are warnings, not errors:

MessageWhat it means
pfctl: Use of -f option, could result in flushing of rules… Normal advisory — pf loaded your rule. Safe to ignore.
No ALTQ support in kernel / ALTQ related functions disabled macOS does not include bandwidth-shaping (ALTQ). Has no effect on port forwarding.
pfctl: pf already enabled The packet filter was already running — this is fine. The -e flag is harmless when pf is already active.
✅ If the command exits without an error (just warnings above), the rule is active. Test it by visiting http://reconnect.zoho.tool in your browser — no port needed.
🔄 This rule is temporary — it clears on reboot. The app itself keeps running on port 9898 unchanged. Re-run the command after each reboot, or add it to /etc/pf.conf to make it permanent.

Option B — Run on port 80

bashpm2 delete reconnect
sudo PORT=80 pm2 start npm --name reconnect -- start
pm2 save
⚠️ macOS caveat: port 80 requires root. Running PM2 as root is not recommended. Prefer Option A.

Option C — Just use a bookmark (simplest)

Bookmark http://reconnect.zoho.tool:9898 and name it Reconnect. One click — you never type the port again.

Authentication (OTP / Zero-Trust)

🔐 OTP is the default and recommended flow. The legacy password/key flow is a fallback that is being phased out and will be removed in a future release. Switch flows in Settings (⚙).

How OTP works

SSH access is password-less and certificate-based through the org's zero-trust gateway (ZAC). You never store or type a server password — an emailed one-time passcode authorizes each login.

  • 1 You click Connect; the session is intercepted by 0Agent and tunnelled via ServiceEdge to the AppConnector in the target DC.
  • 2 The AppConnector enforces department/access policy, then prompts for an OTP (REConnect shows a passcode modal).
  • 3 Enter the OTP from your Zoho email. The AppConnector gets a short-lived ZAC-signed certificate and connects you to the target server.
flowYou ─▶ 0Agent ─▶ ServiceEdge ─▶ AppConnector ──(OTP + dept policy)──▶ ZAC (signs cert)
                                        └────────────(cert auth)────────────▶ Target SSH server
🛡️ The gateway allows one interactive shell per login and blocks SFTP, SCP, port forwarding, X11 / agent forwarding at the network layer — even if enabled on the target.

What that changes in REConnect

CapabilityLegacy modeOTP mode
ConnectDirect, stored password/keyTunnel + emailed OTP (no stored credential)

Prerequisites for OTP access

  • ✓ Policy / ZService access — request via the PAM support channel, or have your team DRI email pam-support@zohocorp.com (CC your manager) with your email, ZService name(s), and department. Wait for approval.
  • ✓ 0Agent network — set the default network to the right zero-host (e.g. CT1 Localzoho Network) and reload policies; revert to none when done.
  • ✓ Verify first — ssh sas@<server-ip> should prompt for an OTP in a plain terminal.

Common issue — “Host key verification failed”

If SSH refuses to connect because the server's key changed, remove the stale entry and reconnect (affects only that host):

bash# replace with your server IP
ssh-keygen -R <server-ip>

Security Notes

ConcernHow Reconnect Addresses It
Credential storage SSH passwords and private keys are encrypted with AES-256-GCM before being written to SQLite. The encryption key is derived from a .secret file generated on first run.
Application access Protected by APP_PASSWORD; all routes check a signed session cookie. In development, omitting APP_PASSWORD bypasses the login screen for convenience.
Network exposure Server binds to 127.0.0.1 by default — not reachable from other machines unless you explicitly change HOST.
Session management Sessions are signed with a random key; tampering with the cookie invalidates it immediately.
SFTP operations File reads and writes go through the SSH connection — no additional ports need to be opened.
